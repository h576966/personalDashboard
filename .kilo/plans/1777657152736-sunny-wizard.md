# Plan: Iteration 2 — Cache, Quality Filtering, Time Range

## Context

MVP is built and working:
- Next.js 16 App Router, TypeScript, Tailwind v4
- `src/app/api/search/route.ts` — POST handler with input validation and Brave API integration
- `src/lib/brave.ts` — Brave Search API client (web endpoint only)
- `src/lib/search/` — pipeline: filter (empty/dedup) → score (term matching + short desc penalty) → sort
- `src/app/page.tsx` — search form + results display (title, url, description, score badge)

## Architecture (unchanged)

```
Browser (search form) → POST /api/search → route handler
  → [NEW: cache check] → lib/brave.ts (if miss)
  → lib/search/ (blocklist pre-filter → existing filter → quality-enhanced score → sort)
  → [NEW: cache store] → JSON response → Browser (render results)
```

## Features

### 1. Cached Search Results (Rate Limit Safety)
- **TTL**: 60 seconds
- **Storage**: In-memory `Map<string, CacheEntry>` on the server
- **Cache key**: `${query}|${count}|${freshness || "any"}`
- **Max entries**: 500 (with periodic cleanup of expired entries every 5 minutes)
- **Source**: SearXNG approach (instance-level in-memory caching to avoid hitting upstream rate limits)

### 2. Result Snippet Quality Filtering
- **Approach**: Domain boost/penalty + heuristics (per user choice)
- **Low-quality results**: Removed entirely (not just scored down)
- **Curated domain lists** in a new `quality.ts` module:
  - `BLOCKED_DOMAINS` — content farms, clickbait mills (pinterest.com, quora.com, buzzfeed.com, etc.)
  - `BOOSTED_DOMAINS` — reputable sources (.edu, .gov, wikipedia.org, github.com, stackoverflow.com, mdn, etc.)
- **Heuristics added to scoring**:
  - Rich description bonus: > 100 chars (+1), > 200 chars (+2)
  - Short/generic title penalty: < 10 chars (-1)
  - Deep URL path penalty: 5+ path segments (-1)

### 3. Time-Range Filtering
- **Brave API param**: `freshness` — `pd` (past day), `pw` (past week), `pm` (past month), `py` (past year)
- **Default**: No filter (any time)
- **UI**: Horizontal preset button strip between header and search form
  - Buttons: `Any time | Past 24h | Past Week | Past Month | Past Year`
- **Validation**: Only accepted values + `undefined`; unknown values → 400 error

### Step 4: Create in-memory cache module
- **Files**: `src/lib/cache.ts` (create)
- **Action**:
  - Export `searchCache` object with `get(key)`, `set(key, results)`, `clear()` methods

## Risks / Edge Cases

| Risk | Mitigation |
|------|------------|
| `setInterval` double-start in Next.js dev mode (HMR reloads modules) | Module-level flag `let cleanupStarted = false` — only start interval once |
| Cache grows unbounded (user searches hundreds of unique queries rapidly) | MAX_SIZE cap of 500 entries, oldest evicted on overflow; periodic cleanup of expired |
| Domain wildcard matching false positives (e.g., `notpinterest.com` blocked by `pinterest.com`) | Use suffix match on full domain label boundaries, not substring |
| `extractDomain()` fails on malformed URLs | Return empty string — `isBlocked("")` returns false (safe default) |
| Brave API `freshness` param not documented for free tier | Docs confirm `freshness` is available on all tiers; if 400, catch and surface |
| Post-holiday stale cache (user opens app after days) gets 1st result from cache then 2nd from API | Acceptable — 60s TTL means stale-at-most 60s, and periodic cleanup removes expired |
